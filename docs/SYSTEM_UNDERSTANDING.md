# HinH (Hackathon in Hackathon) 系统架构与代码理解报告

## 1. 项目概述

**HinH (Hackathon in Hackathon)** 是一个基于 AI 驱动的黑客松模拟平台。系统通过多智能体（Multi-Agent）技术，模拟真实黑客松比赛中的团队组建、项目讨论、产品开发（商业计划书撰写）和评委答辩四个阶段。用户输入初始创意后，系统会自动将预设的 AI 角色分组，并驱动这些角色进行沉浸式的多轮对话和任务协作。

该项目采用了典型的前后端分离架构：
- **前端**：基于 Next.js (React) + Tailwind CSS + Zustand 构建，采用《铁拳 8》（Tekken 8）风格的赛博朋克视觉设计。
- **后端**：基于 NestJS + TypeORM + PostgreSQL 构建，负责核心的 Agent 编排、大模型（LLM）对话控制、DuckDuckGo 搜索工具集成以及通过 Server-Sent Events (SSE) 向前端推送实时状态。

## 2. 系统核心模块解析

### 2.1 领域模型与阶段流转 (Phase Execution)

整个模拟过程被严格划分为四个阶段，核心逻辑封装在 `PhaseExecutor` 和 `SimulationService` 中：

1. **Phase 0: 组队阶段 (Grouping)**
   - **逻辑**：非 LLM 调用的纯算法分配。系统读取预设的 40 个角色（分为软件和硬件赛道），根据角色的 MBTI 性格和技能自动分配黑客松岗位（产品经理、前端、后端、设计师、运营）。
   - **分配**：通过计算“领导力得分”选出队长，并按照 5 人一组的规模进行轮询分组，同时将用户输入的创意分配给各组。

2. **Phase 1: 自由讨论 (Discussion)**
   - **逻辑**：队长首先根据分配到的创意提出初步方向，随后组员轮流提出问题。
   - **交互**：所有成员可以选择 1-2 个问题进行回答或补充。在这个阶段，Agent 具备调用外部搜索工具（通过输出 `[SEARCH:关键词]`）的能力，以获取真实市场数据。最后由队长总结出明确的产品名称、核心问题和解决方案方向。

3. **Phase 2: 产品开发 (Development)**
   - **逻辑**：队长根据第一阶段的总结，为每位组员分配商业计划书（BP）的撰写任务。组员确认任务后开始撰写各自负责的板块。
   - **输出**：队长最终将所有内容整合为标准 JSON 格式的 BP 文档，包含项目名、痛点、解决方案、目标用户、核心功能、商业模式和竞争优势。

4. **Phase 3: 评委答辩 (Defense)**
   - **逻辑**：由 `JudgeRunner` 控制。队长向评委团陈述 BP，系统通过 LLM 动态选择 2-3 位最相关的评委进行尖锐提问。
   - **打分**：系统再次通过 LLM 选出最适合回答该问题的团队成员进行作答。最后，所有 6 位评委并行对项目进行 5 个维度（创新性、表现力、完成度、商业潜力、技术难度）的打分并给出点评。

### 2.2 后端架构与并发控制

后端采用模块化设计，核心在于对 LLM 调用的并发控制和 SSE 流式输出：

- **LLM 服务 (`LlmService`)**：封装了 OpenAI 兼容的 API 客户端。为了防止触发 API 速率限制，系统实现了一个基于 `Semaphore` 的并发控制器（默认最大并发为 6）。
- **搜索服务 (`SearchService`)**：实现了对 DuckDuckGo HTML 页面的抓取和解析，同样使用 `Semaphore` 控制并发（默认最大并发为 3），并带有简单的内存缓存机制。
- **Agent 封装 (`Agent`)**：每个角色被实例化为一个 Agent 对象，包含特定的系统提示词（System Prompt）。Agent 在对话中如果输出特定的搜索标记，会被拦截并触发 `SearchService`，随后将搜索结果注入上下文要求 Agent 重新回答，实现了基础的 ReAct 模式。
- **实时通信 (SSE)**：`SimulationService` 维护了一个 `Map<string, Subject<MessageEvent>>`，在模拟运行过程中，任何状态变化（如 `message`, `agent_typing`, `tool_call`, `phase_change`）都会被实时推送到前端。

### 2.3 前端架构与状态管理

前端注重视觉表现和实时状态的渲染：

- **状态管理 (`simulation-store.ts`)**：使用 Zustand 管理全局状态，包括当前的 `simulationId`、`currentPhase`、分组信息、消息列表、评委打分结果以及各个组当前正在打字的 Agent 状态（`typingAgents`）。Store 负责建立和维护 SSE 连接，并解析后端推送的事件流。
- **视觉系统 (`globals.css` & `layout.tsx`)**：深度定制了基于 Tekken 8 的 UI 风格，大量使用粉色（Pink）和青色（Cyan）的霓虹效果、倾斜的卡片设计（Skew）、自定义滚动条以及 CRT 扫描线和粒子背景效果。
- **核心组件**：
  - `RosterGrid`：负责展示角色网格。通过复杂的状态机（revealing -> revealed -> grouping）和 CSS 动画，实现了角色卡片的洗牌、分组和跑马灯滚动效果。
  - `ChatSidebar`：实时渲染当前选中组的对话流，支持 Markdown 渲染，并能优雅地展示 Agent 的打字状态和工具调用（搜索）过程。

## 3. 数据模型 (Database Schema)

系统使用 TypeORM 管理三个核心实体：

- **`Simulation`**：记录一次完整的模拟运行，包含输入的创意数组、生成的分组配置（JSONB）、当前阶段和运行状态。
- **`Message`**：扁平化存储所有的对话记录，包含所属的模拟 ID、组 ID、阶段、发言 Agent 信息以及文本内容。
- **`Result`**：存储最终的评审结果，包含 BP 文档（JSONB）、评委打分详情（JSONB）和计算出的总分。

## 4. 总结与迭代建议

该系统是一个结构清晰、完成度较高的多智能体协作演示平台。核心的业务化流转逻辑非常严密，前后端的 SSE 通信机制也相对稳健。

在后续的迭代开发中，可以考虑从以下几个方面入手：
1. **Agent 能力扩展**：目前的工具调用仅限于搜索，可以增加代码执行、图表生成等工具，使产品开发阶段（Phase 2）产生更具象的产物。
2. **交互性增强**：允许人类用户在 Phase 1 或 Phase 2 中途介入，作为“投资人”或“导师”对方向进行微调。
3. **模型解耦与优化**：针对不同的任务（如提问、打分、总结）可以配置不同量级的模型，以平衡成本和推理质量。

我对目前的系统架构已经有了深入的理解，随时可以基于此基础开始具体的迭代开发工作。
